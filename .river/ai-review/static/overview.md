# AIレビュー運用 概要（Static）

## 目的

- River-Reviewer のレビュー運用に AIレビューを統合し、レビューの一貫性と速度を上げる。
- ノイズを抑えつつ、要約と指摘の質を安定化する。
- 大規模 PR のコストとレビュー負荷を制御する。
- セキュリティとデータ取り扱いを明文化する。

## 用語整理（SaaS 版 / OSS 版）

- **SaaS 版**: AIレビューサービスを GitHub アプリとして導入し、Webhook/Event でレビューが動く方式。
- **OSS 版（GitHub Actions）**: PR レビュー用の GitHub Actions を自前で動かす方式。モデル選択の自由度は高いが、保守とセキュリティ責任が増える。

## 導入目的と評価軸

- **レビュー一貫性**: 指摘観点のブレ削減、レビュー粒度の標準化。
- **ノイズ抑制**: 重要度の低いコメントを抑え、読みやすさを維持できるか。
- **要約の有用性**: 変更点の俯瞰整理がレビュー効率に寄与するか。
- **コストと速度**: PR あたりのコスト、レビュー応答の遅延の許容度。
- **オンボーディング支援**: レビュー観点の言語化や共有が進むか。
- **セキュリティ**: どのデータがどこへ送られるか、社内ルールと整合するか。

## 期待値調整（必須）

- AI レビューは万能ではない。誤検知・見落としが起きる前提で、人間が最終判断する。
- **詳細レビューより要約の ROI が高い場面が多い**。特に大規模 PR では要約の価値が高い。
- “指示を書けば完全に制御できる”わけではない。設定フラグと運用ルールを優先する。

## 推奨デフォルト（Static）

- **要約優先**: 変更の全体像・レビュー観点の整理に重点を置く。
- **ノイズ抑制優先**: LGTM だけのコメントや小変更レビューは抑制する。
- **重要領域優先**: security/infra/core など重要パスは深掘り対象にする。

## 運用例

- 大規模 PR では AIレビューは要約中心、River-Reviewer は重要ファイルのみ深掘り。
- 小変更や機械生成差分では AIレビューの詳細コメントを無効化し、要約だけ残す。

## 補足（SaaS/OSS の選択指針）

- OSS 版がアーカイブやメンテ薄い場合は **SaaS 版を第一候補**とする。
- OSS 版を続ける場合は **社内フォーク固定 + 依存更新責任**を明示する。

## 参考（運用上の示唆）

- AI レビューは「レビュー観点の標準化」や「オンボーディング支援」に効果がある。
- 変更要約の生成は、詳細指摘よりレビュー効率に寄与しやすい。
