## What worked

1. **Codify-then-validate caught its own gap in the same session.** #898 codified the Stop rule but missed reopen conditions → #902 added them. The rule's own "3 failure scenarios" question would have caught it pre-merge; that gap drove A4 (#904 Self-Review Checklist), which then prevented a similar gap in #911 Phase 2/3 design.
2. **Multi-perspective review (Self / Codex / Gemini) found convergent defects.** Concurrency miss, default coupling, Stop-rule reopen all surfaced via the 3-perspective fan-out, fixed in #902. Single-perspective review missed all three.
3. **Stop-on-Stop-Judge held under pressure.** "自律的に進めて" repeated 8+ times with no new scope → Codex re-judged Stop every time, the agent did not silently override. The rule's "re-judgment trigger, not auto-release" wording was load-bearing.
4. **`workflow_dispatch` + local script duality.** #900 / #902 / #907 evolution: workflow has its uses, local script has its uses, project policy lands on local-first (#926). Choosing both means the right tool is always at hand.
5. **"Posture, not progress" guard.** Hand-written goldens were rejected multiple times. Each community-skill slice (#905 / #909 / #919) shipped scaffolding only; goldens deferred to real `promptfoo eval`. #924 + #926 codified the run path.
6. **Issue carve-out as graceful close.** #868 closed by reframing "incorporation" (done) vs "quality gate" (separate work, #929). #911 same pattern (Slice C → #921). Both kept the parent issue clean and the next work scoped.

## What didn't work

1. **#906 was a same-session defect that 3-perspective review missed.** GITHUB_TOKEN no-recursion is a documented GitHub property that none of the reviewers (Self adversarial, Codex, Gemini) flagged on #902. Mitigation: #907 fixed it; the underlying lesson is that "Code-gen self-review" needs an explicit "GitHub Actions privileges" line item, not just generic concurrency / failure-mode questions.
2. **`gh auth` account switching keeps biting.** "kominem-unilabo does not have permissions" error fired 3+ times mid-session. The switch is required before every merge; not automated, not aliased.
3. **AGENTS.md pollution kept happening.** Every session-start `git checkout AGENTS.md` was needed; the `claude-mem` hook re-adds context blocks. No fix attempted (W2 from prior retro stays "対症療法" status).
4. **Action dist rebuild was forgotten once.** #917 shipped with `Action dist freshness` failing because the bundled action JS was not rebuilt for the `--ensemble` flag change. Fixed quickly, but the gap is in tooling: there is no automatic post-commit hook that rebuilds the dist when `src/cli.mjs` changes.
5. **CI flake / release-please choreography ate cycles.** Most `chore(main): release` PRs needed at least one local kick because the workflow's GITHUB_TOKEN cannot fire downstream CI (#906). The fix is documented (#907 + #926) but the **operational pattern itself**—kick, wait, update-branch, merge—still happens per release.

## Surprises

- **#868 was already "complete" by literal-title reading.** The issue body said "incorporation". 6 skills were incorporated. The `recommended: true` flip was a separate concern that the original issue did not gate completion on. Spending real LLM-eval cycles inside this session would have stretched #868 beyond its title—carve-out (#929) was the right move and unblocked closing.
- **Codex was the brake more than the accelerator.** Across ~20 consultations, ≥10 ended in "Stop continue" or "narrow the scope". Without that brake, the session would have produced 35+ PRs (per prior retro's profile) and probably re-introduced the same "stop condition weakness" defect.
- **A goal-hook (`/goal`) is a useful tool.** Closing #868 needed a session-scoped condition to force interpretation; without it the same input ("#868 を完了") would have been re-judged Stop every turn.

## Stop-on-Stop-Judge—stress-test results

The rule held. Every autonomous-progression prompt without new scope or new information was re-judged Stop. The "self-violation" moments were:

- **One soft violation**: presenting curated next-step lists in response to status checks, which nudges the user to re-prompt. Acceptable in moderation; #904's "operating mode" table covers it.
- **One legitimate override**: #906 discovery during release flow was real new information; reopen-condition #1 fired; #907 shipped. Rule worked as designed.
- **One borderline**: the `/goal #868` hook overrode Stop. That's by design (`/goal` is the user explicitly setting a session-scoped condition), but it is also a reminder that any external "must complete X" signal trumps the rule—that's not a bug; it's the rule's intended interface.

### What stayed the same and what changed

- **Same**: "対応を進めて" is a re-judgment trigger, not a release condition.
- **Changed (#904 codified W6)**: Stop phase now has an explicit operating mode table—status reports are OK, retrospectives are encouraged, curated-list nudging is a soft violation.
- **New gap**: re-judgment fatigue (consulting Codex 5+ times with the same answer). A9 addresses it.

The retrospective above was written mid-session (#930). The remaining work that

### Durable lesson: "completed" report ≠ wired-through

A2-3-runners (#933) discovered that `riskAssessment` was computed in
`buildExecutionPlan` but **never returned**—so every `plan.riskAssessment`
consumer silently received `undefined`. The prior retrospective listed
A2-fix-4 (#877) as having closed this silent-skip. It had not: the field was
forwarded at the _call site_ (`generateReview` received `plan.riskAssessment`)
but the _source_ never emitted it. The forwarding was wired to a value that
was always undefined.

**Why it slipped**: the #877 work added the consumer side and a test that the
consumer _forwards_ the value, but no test asserted the _producer_ emits it.
A "forwards X" test passes whether X is a real value or `undefined`.

**Rule (new, P1 for next sprint)**: a silent-skip fix is not complete until a
test asserts the value is **returned by the producer AND consumed by the
consumer**—not just "forwarded if present". The `review-runner-snapshot-field`
test (#933) now covers the producer side for these four fields; back-fill the
same assertion shape for any future forwarded field.

### Delivery shape that worked

The #878 epic shipped as 4 independently-mergeable slices following the
design doc (#910): schema → todo-marked failing test → producer → consumer.
The todo-marked test (#927) acted as an executable contract; flipping
`{ todo: true }` to a real test in the impl PR (#935) was the objective
"done" signal. Recommend this pattern for future multi-slice features.

### Design-only discipline held twice

Both #910 (A2-3 design) and #938 (cost control design) shipped as design docs
with explicit implementation-hold conditions rather than speculative code. The
cost-control doc (#921) went further: it named telemetry—not the workflow —
as the real first deliverable, because the 3-mode branching cannot be validated
without cost data. This is the prior retro's "stop condition" discipline
applied at design time.
