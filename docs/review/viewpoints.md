# レビュー観点（エージェント向け）

> **出典**: [pages/reference/review-policy.md](../../pages/reference/review-policy.md) Section 1.2 + 4.x
> エージェント・プロンプトから参照するための内部向けフラットチェックリスト。人間向けの詳細は出典を参照。

## 共通観点

| 観点           | チェック内容                                       |
| -------------- | -------------------------------------------------- |
| 可読性         | 命名の適切さ、構造の明確さ、意図が読み取れるか     |
| 拡張性         | 将来の変更・機能追加に対する柔軟性                 |
| パフォーマンス | 実行効率、リソース使用、スケーラビリティ           |
| セキュリティ   | 脆弱性、データ保護、認証・認可の適切さ             |
| 保守性         | デバッグのしやすさ、テストカバレッジ、ドキュメント |

## フェーズ別の追加観点

### Upstream（設計）

- アーキテクチャ決定との整合性
- ADR との照合
- 設計意図の明確さ
- インターフェース設計の適切さ

### Midstream（実装）

- コード品質と可読性
- 命名規則とスタイルガイドの遵守
- エラーハンドリングの適切さ
- 重複コードの削減

### Downstream（テスト/QA）

- テストカバレッジ（正常系/異常系/境界）
- エッジケースのテスト
- テストの可読性と保守性
- テスト実行時のパフォーマンス

### ドキュメント変更（doc-only / #1068）

差分が `*.md` / docs 配下のみの PR では、コード観点（特に Downstream のテスト）が空振りし、ドキュメント固有の実害を拾えない。以下の専用観点でレビューする:

- **リンク切れ**: 内部相対パス・アンカー・外部リンクが有効か
- **正本整合**: 設定ファイルや実装と矛盾する記述がないか（例: 設定値とドキュメント記載の乖離）
- **版数・件数・日付のずれ**: 実態と一致しているか
- **実行例の到達性**: 記載したコマンド・エンドポイント・パス・オプションが実在し到達可能か
