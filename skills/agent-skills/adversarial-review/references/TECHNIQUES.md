# Adversarial Review Techniques（敵対的レビュー手法）

## 理論的背景

敵対的レビューは、Daniel Kahneman の「事前検死（Pre-mortem）」、軍事戦略の「ウォーゲーム」、
Karl Popper の「反証主義」に基づく、認知バイアス対策のフレームワークである。

共通原理: **「自分に都合の悪い情報を体系的に引き出す」**

---

## 手法1: Pre-mortem（事前検死 / 死亡前死因分析）

### 起源

Gary Klein が提唱した意思決定手法。プロジェクト開始前に「失敗した」と仮定し、
その原因を逆算することで、楽観バイアスを排除する。

### 対策するバイアス

- **楽観バイアス**: 「うまくいくはず」という根拠のない確信
- **生存バイアス**: 成功例だけを見て失敗パターンを無視する傾向
- **計画錯誤**: 最良のシナリオを基準に計画を立てる傾向

### コードレビューでの適用

```text
従来: 「このコードは正しいか？」
Pre-mortem: 「このコードが本番で障害を起こしたとしたら、原因は何か？」
```

変更の「正しさ」ではなく「脆さ」に注目することで、見落としがちなエッジケースや
暗黙の前提が浮上する。

### 効果的な問いかけ

- 「この変更が6ヶ月後にインシデントを引き起こすとしたら？」
- 「この前提が崩れるのはどんな状況か？」
- 「このシステムで最初に壊れるのはどこか？」

---

## 手法2: War Game（ウォーゲーム / 敵対的シミュレーション）

### 起源

軍事戦略における「レッドチーム」の概念。味方の計画に対して、
敵の立場で攻撃シナリオを構築し、防御の穴を発見する。

### 対策するバイアス

- **自己中心バイアス**: 自分の視点だけでシステムを評価する傾向
- **正常性バイアス**: 「攻撃されないだろう」という思い込み
- **フレーミング効果**: 開発者視点のフレームで脅威を過小評価する傾向

### コードレビューでの適用

```text
従来: 「このAPIは仕様通り動くか？」
War Game: 「攻撃者がこのAPIを悪用するとしたら、どう攻める？」
```

### 攻撃者ペルソナの活用

異なるペルソナを設定することで、多角的な脅威分析が可能になる：

| ペルソナ             | 動機                     | 典型的な攻撃                   |
| -------------------- | ------------------------ | ------------------------------ |
| 外部攻撃者           | データ窃取、サービス妨害 | 認証バイパス、インジェクション |
| 認証済み悪意ユーザー | 権限昇格、他者データ閲覧 | IDOR、パラメータ改ざん         |
| 内部脅威者           | 情報漏洩、バックドア     | 設定改ざん、ログ改竄           |
| 自動化ボット         | 大量悪用、リソース消費   | レート制限回避、スクレイピング |

### 効果的な問いかけ

- 「このエンドポイントを壊す最も簡単な方法は？」
- 「認証されたユーザーが他者のデータにアクセスする方法は？」
- 「このシステムに1万リクエスト/秒を送ったら何が起きる？」

## 手法3: Logic Torturing（論理の拷問 / 反証的検証）

### 起源

Karl Popper の反証主義と、哲学的議論における「Devil's Advocate（悪魔の代弁者）」
の手法を組み合わせたもの。

### 対策するバイアス

- **確証バイアス**: 自分の判断を支持する情報だけを集める傾向
- **アンカリング**: 最初の判断に引きずられる傾向
- **サンクコストの誤謬**: 既に投資した選択を変更しにくい傾向

### コードレビューでの適用

```text
従来: 「この実装は要件を満たすか？」
Logic Torturing: 「この設計判断の論理的な穴はどこか？前提が変わっても成立するか？」
```

### 5つの検証レンズ

1. **前提の検証**: 暗黙の仮定を明示化し、その妥当性を問う
2. **代替案の検証**: 棄却された選択肢の理由を問う
3. **境界条件の検証**: 極端な条件下での挙動を問う
4. **整合性の検証**: 既存設計との矛盾を問う
5. **可逆性の検証**: 判断を覆すコストを問う

### 効果的な問いかけ

- 「なぜこの方法で、他ではないのか？」
- 「この判断が間違いだとわかったとき、元に戻せるか？」
- 「この前提が3ヶ月後も成立する保証は？」

---

## 3手法の使い分け

```text
変更の種類を判定
│
├─ 設計・アーキテクチャ変更
│   └─ Pre-mortem（主） + Logic Torturing（副）
│      「この設計が失敗する原因は？」「この判断の根拠は十分か？」
│
├─ セキュリティ関連変更
│   └─ War Game（主） + Logic Torturing（副）
│      「攻撃者ならどう悪用する？」「この防御策の前提は正しいか？」
│
├─ 重要な技術判断を含む変更
│   └─ Logic Torturing（主）
│      「この選択の論理に穴はないか？」
│
└─ 大規模変更（高リスク）
    └─ 全手法実行
       「失敗シナリオは？」「攻撃経路は？」「論理の穴は？」
```

## アンチパターン

以下は敵対的レビューの誤用であり、避けるべきである：

1. **批判のための批判**: 建設的な改善提案なしに穴だけを指摘する
2. **完璧主義の強制**: すべてのリスクを排除しようとする（リスクゼロは不可能）
3. **著者への攻撃**: 判断の穴を個人の能力の問題として扱う
4. **パラリシス**: 指摘が多すぎてアクションが取れなくなる
5. **一般論の羅列**: 差分に紐づかない教科書的なリスク列挙

## 原則

- **目的は判断の強化であり、否定ではない**
- **差分に紐づく具体的な指摘のみ**
- **指摘には必ず次のアクションを添える**
- **最大5件に絞り、最も重大なものを優先する**
